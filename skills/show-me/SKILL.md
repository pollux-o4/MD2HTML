# show-me

repo 의 Markdown 을 사용자 쿼리에 맞춰 인터랙티브 HTML 로 큐레이션한다. 자세한 spec 은 사용자 repo 의 `docs/md-show-me-prd.md` + `docs/md-show-me-milestone-1..5-*.md` 참고 (있을 때).

## 동작 흐름

### 1) 직전 manifest 확인 (M5 lazy stale check)

slug = 쿼리에서 url-safe slugify (예: `"리뷰 정리"` → `리뷰-정리`).

`<repo>/.agent-output/show-me/reports/<slug>/manifest.json` 이 있으면 그 slug 의 *직전 호출* 이 있었던 것. 같은 slug 의 `_history/<ISO-timestamp>/manifest.json` 중 가장 최근과 *현재 repo 의 .md 파일 mtime+size* 를 비교 → manifest diff (added / removed / changed sources).

처음 호출이면 직전 manifest 없음. diff 단계 skip.

### 2) 스캔 + 큐레이션 (M1 discovery)

**sub-agent 안에서** 다음 수행 (parent context 보호):

1. **후보 발견** — Glob/Grep 으로 trusted folder 의 `.md` 검색. 쿼리 의도에 맞춰 즉석. 사전 path list sequence 없음 — LLM 이 training data 의 관례 (`docs/adr/`, `docs/references/` 등) + 쿼리 키워드로 판단.
2. **기본 exclude 적용**:
   - dependency: `node_modules/**`, `.venv/**`, `__pycache__/**`, `*.egg-info/**`
   - generated/build: `dist/**`, `build/**`, `coverage/**`, `.agent-output/**`, `.next/**`, `.nuxt/**`, `target/**`, `vendor/**`
   - tool worktree: `.claude/worktrees/**`, `.claude/scratch/**`
   - `.gitattributes` 의 `linguist-generated=true` 경로
   - 본문 marker: `AUTO-GENERATED`, `DO NOT EDIT`, "Generated by", frontmatter `auto_generated: true` (LLM 판단)
3. **옵션 config** — repo root 의 `.show-me.toml` (있으면) 의 `trusted_sources`, `role_patterns`, `exclude_patterns`, `enable_cdn_libs` 등 읽고 default 덮어쓰기.
4. **head-preview 추출** — 후보별로 filename + H1 + frontmatter (있으면) + 본문 첫 5~10줄 + 옵션 `##` heading.
5. **큐레이션** — preview 만 보고 LLM 이 *full Read 할 N개* 선택 (보통 3~10, **hard cap 없음**). preview set 이 부족하면 agent loop 로 더 fetch. preview 가 sub-agent context 한도 넘으면 페이지 단위 multi-turn.
6. **full Read** — 최종 N개 full Read → M2 입력.

각 후보에:
- `reason`: LLM 의 자연어 한 줄 ("title 과 본문이 '리뷰' 워크플로우를 직접 다룸" 식)
- `abstract`: frontmatter `abstract` 있으면 그것, 없으면 head 5~10줄
- `role`: LLM 추론 (handoff / draft / decision / reference / working-memory / general)
- `mtime`, `size`: stat 결과
- `major_headings`: `##` 수준 헤딩 목록 (M5 drift 감지용)

**unknown_markdowns**: trusted folder 안인데 role 분류 못 한 `.md` 들을 별도 리스트로 manifest 에 기록.

### 3) Manifest 작성 (M1 output)

```json
{
  "query": "<원본 쿼리>",
  "created": "<ISO timestamp>",
  "sources": [
    {
      "path": "docs/example.md",
      "title": "Example",
      "abstract": "...",
      "role": "review",
      "reason": "...",
      "major_headings": ["문제", "결론"],
      "mtime": "2026-05-27T14:30:00",
      "size": 1234,
      "updated": "2026-05-27"
    }
  ],
  "unknown_markdowns": [
    {"path": "docs/scratch/idea.md", "reason": "role 매칭 안됨"}
  ]
}
```

### 4) 인터랙티브 HTML 생성 (M2)

질의 형태에 맞춰 LLM 이 매번 새로 생성. 같은 source set 이라도 query 다르면 다른 HTML.

**Baseline 3 (모든 호출에 strict 포함)** — `assets/snippets/` 참고:
- `n1-answer-card.html` — citation 포함 답변 카드 (상단)
- `n5-back-link-badge.html` — 각 인용 옆 원본 .md 링크 (Windows 경로 `/` 로 통일)
- `n4-copy-as-prompt.html` — 모든 인터랙션 옆 `[Copy]` / `[Save]` 오버레이

**선택 베이스라인** (LLM 적합성 판단):
- N+1 discovery transparency 패널 — 첫 사용 / dirty repo 결과 의심 시 가치 ↑
- N33 원문 주석 레이어 — 본문 review 흐름일 때 가치 ↑

**기본 CSS**: `assets/base.css` 를 inline 로 출력 HTML 의 `<style>` 에 포함.

**패턴 선택**: 쿼리 + source data shape (timestamps / cross-refs / hierarchy / tidy table / alternatives / parameterized concept) × 시각화 표면 (timeline / graph / gallery / table / diff 등) 으로 매트릭스 매핑. 자세한 추천은 사용자 repo `docs/research/persona-evaluation-nadia-park.md` §5-1.

**41 패턴은 strict 카탈로그 아니다** — 쿼리에 더 맞는 패턴이 있으면 즉석 생성 OK.

### 5) CDN 라이브러리 (필요 시 per-library confirm)

LLM 이 "Tailwind / Vega-Lite / ECharts / Chart.js / Mermaid 등 라이브러리 쓰면 결과가 더 좋아진다" 판단:

1. `.show-me.toml` 의 `enable_cdn_libs` 리스트에 그 라이브러리 있는지 확인
2. 있으면 → CDN `<script>` / `<link>` 태그 inject, 사용
3. 없으면 → **사용자에게 prompt**:
   ```
   이 결과에 <Library> (CDN, ~XX KB) 를 쓰면 더 보기 좋아짐.
   효과: <구체적 설명>
   허용? (y/n, y → .show-me.toml 에 추가, 앞으로 자동 사용)
   ```
4. `y` → `.show-me.toml` 에 atomic append, 이번 호출에 사용
5. `n` → inline SVG/CSS/vanilla JS 로 대체, 다시 묻지 않음 (declined 기록)

기본은 **inline only**. 외부 호출 0건.

### 6) 출력 + history 보존

출력 위치: `<repo>/.agent-output/show-me/reports/<slug>/`

새 호출 시:

1. 기존 `index.html` + `manifest.json` 이 있으면 → `_history/<ISO-timestamp>/` 로 이동 (atomic rename)
2. 새 결과를 `index.html` + `manifest.json` 에 atomic write (temp → rename)

```
reports/<slug>/
  index.html              ← 최신
  manifest.json           ← 최신
  _history/
    2026-05-27T10-00-00/index.html
    2026-05-27T10-00-00/manifest.json
    ...
```

timestamp 는 Windows safe (`:` 안 씀, `-` 사용).

skill 은 **자동 prune 안 함** — D1 (repo 망가뜨리지 않음).

### 7) `.gitignore` 처리

첫 실행 (`.agent-output/` 이 `.gitignore` 에 없을 때):

```
⚠️  .agent-output/ 이 .gitignore 에 없음. 자동 추가할까? (y/n)
```

- `y` → `.gitignore` 끝에 `.agent-output/` 한 줄 append (atomic write)
- `n` → 알림만, 다음 호출에 다시 묻지 않음 (`.show-me.toml` 에 `gitignore_prompt_declined = true` 기록)

## Copy-as-prompt (M3)

HTML 안 모든 인터랙션 (slider, drag, toggle, 코멘트, 비교 선택) 옆에 두 버튼:

- `[Copy]` — 클립보드 + toast 알림 ("✅ 복사됨")
- `[Save]` — `SAVE_TO_REVIEW` marker 가 붙은 텍스트를 클립보드로. 사용자가 Claude 세션에 paste 하면 skill 이 marker 감지 → `<repo>/.agent-output/show-me/_review/<slug>.md` 에 atomic block append.

### `[Save]` 의 agent-mediated 저장 흐름

브라우저는 직접 disk write 불가 (skill 의 zero install 제약). 그래서 `[Save]` 는 *2단계 prompt 패턴*:

1. 사용자가 HTML 의 `[Save]` 클릭 → 클립보드에 marker 포함 텍스트 복사:
   ```
   <!-- SAVE_TO_REVIEW: <slug> -->
   ## Review comment
   source_path: ...
   anchor_quote: "..."
   comment: ...
   created: ...
   <!-- END_SAVE -->
   ```
2. 사용자가 Claude 세션 (어떤 세션이든) 에 paste.
3. **skill (또는 main Claude session) 이 `SAVE_TO_REVIEW` marker 감지** → 다음 동작:
   - slug 추출
   - `<repo>/.agent-output/show-me/_review/<slug>.md` 에 atomic append
   - 직전 block 과 dedup (`source_path + anchor_quote + comment` 비교)
   - 사용자에게 toast/알림: `✅ 저장됨 → _review/<slug>.md (총 N건)`

### Marker 감지 룰 (Claude 세션이 알아야 할 것)

사용자 input 에 `<!-- SAVE_TO_REVIEW: ... -->` 패턴 발견 시:
- 슬래시 명령 없어도 자동 처리
- `_review/<slug>.md` 파일이 없으면 생성, 있으면 끝에 `---` 구분자 + block append
- 같은 호출에 marker 가 여러 개면 순차 처리

### `[Save]` append + dedup 규칙

slug 당 1개 파일 누적 (`<slug>.md`). 매 save 마다 새 파일 만들지 않는다.

각 save 의 block 형식 (Markdown):

```markdown
## Review comment
source_path: docs/auth.md
heading: 토큰 만료 정책
anchor_quote: "토큰은 1시간 후 만료된다."
weak_anchor: false
comment: 1시간 짧음. 24시간으로?
created: 2026-05-27T14:30:00+09:00

---
```

**Dedup**: 직전 block 과 `source_path + anchor_quote + comment` 동일하면 append X, "이미 저장됨" toast.

**Toast 형식**: `✅ 저장됨 → _review/리뷰.md (총 N건)` — 누적 건수 표시.

리뷰가 아닌 인터랙션 (slider 값 등) snapshot 도 같은 파일에 block 으로 누적, 필드는 LLM 자유 (`parameter`, `value`, `surrounding_context` 등).

## Stale 감지 시각화 (M5)

manifest diff 결과:

- **added_sources / removed_sources / changed_sources** → HTML 상단 *manifest diff viewer* 영역 (접힘 default, 사용자가 펼침)
- **heading 깨짐** → HTML 상단 *lazy stale banner* (조용히, modal/차단 X). 사용자가 클릭하면 detail 펼침
- **`.show-me.toml` 또는 `.gitattributes` 변경** → 다음 호출에 새 규칙 자동 적용 (별도 rebuild 단계 없음)

## 출력 글 톤 (항상 켜진 룰)

도구가 만들어내는 HTML 본문은 아래 5룰을 무조건 따른다. 사용자가 한국어 학습자 / 비전공 / 시각 학습자라는 가정.

1. **영어 단어 음차 무분별 섞기 금지** — "baseline", "anti-pattern", "jargon", "context", "framework" 같은 단어는 대체 가능한 일반어면 한글로 풀이. 대체 불가 전문 용어만 원어 유지 + 첫 등장 시 괄호로 한글 풀이 병기.
2. **어려운 한자어 남발 금지** — "당사는", "본 도구는", "여하간", "차치하고", "수행하다", "활용하다", "구현하다" 같은 표현은 "저희가", "이 도구는", "어쨌든", "제쳐 두고", "한다", "쓴다", "만든다" 로 풀어 쓰기.
3. **번역체 종결어미 금지** — "~에 의해", "~에 다름 아니다", "~이라 하지 않을 수 없다", "~되어집니다" 금지. "~할 수 있어요", "~돼요" 같은 자연스러운 한국어로.
4. **em dash (—) 남발 금지** — 한 단락당 1개 이내. 콤마·괄호로 대체 가능하면 대체.
5. **지식의 저주 차단 (의무)** — HTML 의 모든 전문 용어 (도메인 약어, 3글자 이하 영문, 비전공자가 모를 단어) 는 첫 등장 시 1줄 풀이 의무. 풀이는 괄호 안 또는 hover 툴팁 어느 쪽이든 OK. 같은 페이지 안 동일 용어 재등장은 풀이 생략 (중복 효과 회피).

근거 — 페르소나 H (박서윤, 비CS 풀스택 취준생) 의 "한글 사이 영어 단어 박혀 읽다 멈춤" pain. 학술: `academic-communication-korean.md`, `academic-plain-language.md`, `academic-linguistics-korean.md`, Pinker (2014) *curse of knowledge*.

## 제약 (반드시)

- **자동 패키지 설치, 자동 모델 다운로드 — 금지** (zero install)
- **자동 외부 네트워크 호출 — 금지** (zero network, CDN 라이브러리만 사용자 허용 시 예외)
- **원본 Markdown 자동 편집 — 금지** (Markdown = source of truth)
- 모든 파일 쓰기는 **atomic** (temp → rename, 부분 손상 방지)
- discovery 는 **sub-agent** 안에서 (parent context 보호)
- 출력 HTML 은 server 없이 `file://` 더블클릭으로 열려야 함

## 사용자 repo 의 PRD 와의 관계

`/show-me` skill 의 행동은 사용자 repo 의 `docs/md-show-me-prd.md` (있으면) 의 13개 결정 + Constraints 5개 + baseline 3 + selector rulebook 을 따른다. 이 SKILL.md 는 그 PRD 의 압축이고, 자세한 근거는 milestone 파일들 참고.

PRD 가 repo 에 없는 케이스 (skill 만 가진 사용자) 에서는 이 SKILL.md 가 단독으로 동작 명세 역할.
