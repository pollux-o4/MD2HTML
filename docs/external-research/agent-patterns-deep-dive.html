External Research Deep Dive

리서치 형식, 컨텍스트 파일, 자동 평가 루프는 어떻게 에이전트 품질을 바꾸는가

Pattern 04, 06, 07과 Ethan Mollick의 Claude Sonnet 4.5 경제학 논문 재현 사례를 심화 조사한 보고서입니다. 조사 에이전트가 작성한 Markdown 원본을 별도 검증 에이전트가 확인했고, 이 HTML은 검증 결과를 반영해 표현을 보수적으로 조정했습니다.

작성일: 2026-05-18 원본 MD: agent-patterns-deep-dive-research.md 검증 반영: 날짜·주장 강도·출처 성격 조정

한 문장 결론

산출물 형식

리서치 에이전트에게 형식을 주면 결과의 검토 가능성과 일관성이 올라간다. 단, 형식 자체가 진실성을 보장하지는 않는다.

컨텍스트 파일

CLAUDE.md, AGENTS.md 같은 파일은 에이전트에게 반복 규칙과 작업 경계를 주는 계약서 역할을 한다.

자동 평가 루프

테스트, 벤치마크, 출처 체크, 재현 수치 비교처럼 자동 판정 기준이 있어야 장시간 작업을 맡기기 쉽다.

검증자가 조정한 해석 기준

강하게 말해도 되는 것

공식 문서가 특정 제품 동작이나 권장사항을 직접 설명하는 경우입니다. 예를 들어 Claude Code 문서의 CLAUDE.md 설명, OpenAI agent evals의 traces·graders·datasets 설명, Codex agent loop의 AGENTS.md 로딩 설명은 강한 근거로 볼 수 있습니다.

완화해서 말해야 하는 것

개인 실무 글, 회사 블로그의 생산성 사례, Mollick의 논문 재현 사례는 구체적이지만 일반화에는 한계가 있습니다. 따라서 “해결했다”보다 “특정 조건에서 시연됐다”, “품질이 오른다”보다 “검토 가능성을 높인다”로 표현했습니다.

Pattern 04. 리서치 에이전트는 산출물 형식이 있을 때 더 검토하기 쉽다

핵심은 “좋은 글”을 요구하는 것이 아니라 “검증 가능한 구조”를 요구하는 것입니다.

개념

“조사해줘”라고 하면 에이전트는 읽기 좋은 요약을 만들 수 있지만, 근거가 얼마나 강한지, 반례를 봤는지, 출처가 1차인지 2차인지가 흐려질 수 있습니다. 반대로 “주장 / 근거 URL / 출처 유형 / 주장 강도 / 반례 / 의사결정 권고”처럼 형식을 지정하면 결과가 검토 가능한 자료가 됩니다.

왜 중요한가

형식은 에이전트의 탐색 목표를 좁히고, 사람의 리뷰 비용을 낮춥니다. 다만 근거 문서들은 주로 출력 형식 준수와 파싱 용이성을 뒷받침하므로, “리서치 품질 전체가 자동으로 좋아진다”는 식으로 말하면 과장입니다.

실무 근거

OpenAI 도움말은 원하는 맥락, 결과, 길이, 형식, 스타일을 구체화하라고 권장합니다. Anthropic 프롬프트 문서는 복잡한 지시를 구조화하고 원하는 응답 형식을 명시하는 방식을 다룹니다.

Simon Willison은 병렬 코딩 에이전트를 proof-of-concept 리서치와 코드베이스 설명에 쓰고, 그 설명을 다음 프롬프트의 컨텍스트로 재사용하는 흐름을 공개했습니다.

이 작업 자체의 예

이번 요청은 “개념 설명, 실무 예시, 쉬운 예시, 실패 모드, 프롬프트 템플릿, 출처 표, 주장 강도”를 요구했습니다. 그래서 조사 에이전트는 단순 요약이 아니라 검증 에이전트가 확인할 수 있는 구조화된 Markdown을 만들 수 있었습니다.

고등학생도 이해할 수 있는 예

“조선 시대 조사해줘”라고 하면 친구가 긴 글을 가져올 수 있습니다. 하지만 “왕 이름, 주요 사건, 연도, 근거 링크, 시험 포인트를 표로 정리해줘”라고 하면 바로 공부에 쓸 수 있습니다. 형식은 조사 방향을 정하는 채점표입니다.

실패 모드
  • 형식이 너무 복잡하면 빈약한 근거를 억지로 칸에 넣을 수 있습니다.
  • 표나 JSON을 요구해도 모델이 항상 완벽히 지키지는 않으므로 스키마 검증이 필요할 수 있습니다.
  • 출처 표가 있어도 출처 품질은 별도 검증해야 합니다.
프롬프트 템플릿 
다음 주제를 조사하라: {질문}

출력 형식:
1. 핵심 결론 5문장
2. 주장 표: 주장 / 근거 URL / 출처 유형 / 주장 강도 / 반례
3. 실무 적용 예시
4. 실패 모드
5. 지금 결정 가능한 것과 추가 검증이 필요한 것

규칙:
- 회사 발표, 개인 자기보고, 독립 근거를 구분하라.
- 확실하지 않은 내용은 확실하지 않다고 표시하라.

Pattern 06. 컨텍스트 파일은 에이전트용 작업 계약서다

컨텍스트 파일은 매번 반복해야 하는 규칙을 에이전트가 작업 시작 시 읽게 만드는 장치입니다.

개념

CLAUDE.md, AGENTS.md 같은 파일은 README와 비슷하지만 주 독자가 에이전트입니다. 설치법뿐 아니라 테스트 명령, 수정 금지 파일, 코딩 스타일, 도메인 용어, 실패 시 중단 조건, 승인 필요한 행동을 적습니다.

왜 중요한가

반복 규칙과 성공 기준이 없으면 에이전트는 일반적인 관습으로 추측합니다. 컨텍스트 파일은 그 추측을 줄이고 세션과 에이전트가 바뀌어도 같은 기준을 유지하게 돕습니다. 단, 문서에 적힌 규칙은 강한 힌트이지 완전한 보안 장치가 아닙니다.

실무 근거

Claude Code 문서는 CLAUDE.md가 세션 시작 시 컨텍스트로 로드되고, 프로젝트 아키텍처, 코딩 표준, 공통 워크플로를 담을 수 있다고 설명합니다. GitHub Actions 문서도 저장소 루트의 CLAUDE.md에 코드 스타일과 리뷰 기준을 정의하라고 권장합니다.

OpenAI의 Codex agent loop 설명은 $CODEX_HOME과 작업 경로의 AGENTS.md 또는 override 파일이 입력 컨텍스트에 포함된다고 설명합니다.

실무자 예시

Simon Willison은 MCP부터 설계하기보다 셸 명령과 사용 예시를 AGENTS.md에 적어두는 방식을 제안합니다. Karpathy의 autoresearch도 에이전트에게 program.md를 읽게 하고 수정 범위를 train.py로 제한하는 식의 계약 구조를 보여줍니다.

고등학생도 이해할 수 있는 예

조별 과제에서 “발표 자료 잘 만들어줘”라고 하면 친구가 마음대로 만들 수 있습니다. “10장 이하, 첫 장은 제목, 마지막 장은 참고문헌, 위키백과만 쓰지 말 것, 제출 전 맞춤법 검사”라고 규칙표를 주면 결과가 예측 가능해집니다. 컨텍스트 파일은 에이전트에게 주는 조별 과제 규칙표입니다.

실패 모드
  • 파일이 너무 길면 핵심 규칙이 묻히고 비용과 지연이 커집니다.
  • 오래된 명령이나 폐기된 아키텍처가 남아 있으면 에이전트가 낡은 계약을 따릅니다.
  • 비밀키, 개인정보, 내부 URL을 넣으면 도구 호출 경로에서 노출될 수 있습니다.
  • 삭제, 배포, 결제, 외부 발송은 컨텍스트 파일만 믿지 말고 승인 게이트를 둬야 합니다.
컨텍스트 파일 템플릿 
# AGENTS.md 또는 CLAUDE.md

## Project Contract
- 목적: {한 문장 설명}
- 원칙: 작은 변경, 테스트 우선, 관련 없는 파일 수정 금지
- 수정 금지: {paths}
- 승인 필요: 배포, 데이터 삭제, 이메일 발송, 결제

## Commands
- 테스트: `{test_command}`
- 린트: `{lint_command}`
- 타입체크: `{typecheck_command}`

## Definition of Done
- 관련 테스트 통과
- 변경 파일 요약
- 남은 위험 명시

## Stop Conditions
- 같은 실패가 반복되면 중단
- 요구사항과 기존 규칙이 충돌하면 질문

Pattern 07. 자동 평가 루프가 있으면 장시간 작업을 맡기기 쉽다

에이전트가 오래 일하려면 “잘 됐는지”를 스스로 어느 정도 판정할 수 있어야 합니다.

개념

자동 평가 루프는 수정 -> 실행 -> 측정 -> 유지 또는 폐기를 반복하는 구조입니다. 코딩에서는 테스트, 린트, 타입체크, 벤치마크가 평가 기준이고, 리서치에서는 출처 존재, 주장 강도, 반례 포함 여부가 체크리스트가 됩니다.

왜 중요한가

장시간 작업은 작은 오류가 누적되기 쉽습니다. 자동 평가는 에이전트가 “좋아 보이는 결과”보다 측정 가능한 기준을 향해 움직이게 합니다. 사람이 최종 검토할 때도 무엇을 시도했고 무엇이 통과했는지 추적할 수 있습니다.

실무 근거

Anthropic 평가 문서는 성공 기준을 정의하고 평가를 설계하는 과정을 품질 개선의 중심으로 둡니다. OpenAI agent evals 문서는 traces, graders, datasets, eval runs로 에이전트 워크플로를 반복 측정하고 개선하는 방법을 설명합니다.

Simon Willison은 좋은 테스트 스위트가 코딩 에이전트의 가치를 크게 키운다고 설명합니다.

실무자 예시

Karpathy의 autoresearch는 에이전트가 nanochat 훈련 실험을 반복하게 하는 프로젝트입니다. README 기준으로 5분 훈련 예산, validation bits per byte 지표, 수정 파일 제한을 둡니다. 12 experiments/hour 같은 수치는 자기보고 프로젝트 수치이므로 일반 성능으로 확대하면 안 됩니다.

고등학생도 이해할 수 있는 예

수학 문제집을 풀 때 답지가 있으면 혼자 풀고 채점하고 다시 고칠 수 있습니다. 자동 평가 루프는 에이전트에게 주는 답지와 채점 규칙입니다. 단, 답지가 틀리면 공부 방향도 틀어지듯 평가 기준이 잘못되면 에이전트도 잘못된 목표를 최적화합니다.

실패 모드
  • 테스트가 빈약하면 테스트 통과가 실제 품질을 의미하지 않습니다.
  • LLM judge만 쓰면 같은 모델 계열의 편향이 평가에 들어갈 수 있습니다.
  • 자동 루프에 권한을 너무 많이 주면 비용 폭증, 데이터 유출, 잘못된 명령 위험이 커집니다.
  • 로그, 버전, seed, 실패한 시도가 없으면 결과를 재현하기 어렵습니다.
루프 템플릿 
목표:
{goal}

수정 가능 범위:
{allowed_paths}

평가 명령:
1. `{test_command}`
2. `{lint_or_validation_command}`
3. `{benchmark_or_check}`

루프 규칙:
- 먼저 실패를 재현하라.
- 한 번에 작은 변경만 하라.
- 각 반복마다 명령과 결과를 기록하라.
- 통과하면 멈추고 남은 위험을 보고하라.
- 같은 실패가 3회 반복되면 중단하라.

Ethan Mollick 사례. Claude Sonnet 4.5로 경제학 논문 결과를 재현하게 한 실험

이 사례는 강한 일반 결론이 아니라, 조건이 맞을 때 에이전트가 실제 연구 보조 작업의 일부를 수행할 수 있음을 보여주는 자기보고 사례입니다.

해석 제한

Mollick의 글은 2025-09-29 개인 블로그 글입니다. 독립 벤치마크가 아니며, “AI가 학술 재현 문제를 해결했다”는 뜻이 아닙니다. 기존 데이터와 분석 절차를 다시 돌려 논문 표와 결과가 맞는지 확인하는 reproduction에 가깝고, 새 표본으로 실험을 다시 수행하는 replication과는 구분해야 합니다.

확인된 흐름

Mollick은 Claude Sonnet 4.5에 경제학 논문 텍스트와 replication data archive를 주고, 가능한 범위에서 결과를 재현하라고 지시했습니다. Claude는 파일을 탐색하고 STATA 코드를 Python으로 변환해 결과 재현을 시도했습니다. Mollick은 일부 결과를 spot check했고 GPT-5 Pro를 이용해 별도 모델 교차검토를 했다고 썼습니다.

이 사례가 세 패턴과 연결되는 방식
  • 산출물 형식: 원 논문 수치, 재현 수치, 차이, 실패 원인 표가 있어야 검토할 수 있습니다.
  • 컨텍스트: 논문 본문, 복제 데이터 아카이브, 원 코드, 작업 지시가 에이전트의 작업 계약이 됩니다.
  • 자동 평가: 원 논문 결과와 재현 결과의 차이를 비교하고, 다른 모델 또는 세션으로 교차검토합니다.
고등학생도 이해할 수 있는 예

어떤 친구가 과학 보고서에 표와 계산식을 제출했다고 해봅시다. 에이전트에게 재현을 시키는 것은 그 친구가 제출한 데이터와 계산식을 다시 열어 같은 표가 나오는지 확인하게 하는 일입니다. 새 실험을 하는 것이 아니라, 이미 제출된 기록과 계산이 맞는지 다시 돌려보는 것입니다.

실패 모드
  • STATA와 Python은 결측치 처리, 고정효과, clustered standard errors, factor variable 처리 기본값이 다를 수 있습니다.
  • 복제 패키지 자체가 불완전하거나 데이터 접근 권한이 없으면 에이전트가 해결할 수 없습니다.
  • 파일 크기, 패키지 설치, 라이선스, 실행 시간 제한이 병목이 됩니다.
  • 다른 모델 교차검토는 유용하지만 완전한 독립 감사는 아닙니다.
재현 검토 템플릿 
너는 재현성 검토 보조 연구원이다.

입력:
- 논문: {paper_path}
- replication data archive: {data_path}
- 원 코드 언어: {stata_or_r}
- 목표 실행 언어: {python_or_original}

작업:
1. 핵심 표와 그림 목록을 추출하라.
2. 데이터 아카이브 구조를 요약하라.
3. 원 코드 실행 흐름을 설명하라.
4. 표마다 원 수치, 재현 수치, 차이, 실행 명령을 기록하라.
5. 실패 항목은 data missing, package issue, statistical mismatch, unclear code, resource limit 중 하나로 분류하라.
6. 결론은 재현됨, 부분 재현, 재현 실패, 판정 불가 중 하나로만 표시하라.

- 성공을 과장하지 말라.
- 별도 모델 또는 별도 세션으로 결과표와 코드를 교차검토하라.

실무 운영 모델

단계 에이전트에게 줄 것 사람이 확인할 것
1. Output contract 표, 주장 강도, 반례, 실패 원인, 최종 권고처럼 결과 형식을 먼저 준다. 형식이 결론을 유도하지 않는지, 반대 근거를 담을 칸이 있는지 확인한다.
2. Context contract AGENTS.md나 CLAUDE.md에 명령, 금지 행동, 테스트, 중단 조건을 둔다. 문서가 최신인지, 비밀 정보가 없는지, 권한 게이트가 명확한지 확인한다.
3. Evaluation loop 테스트, 벤치마크, 출처 체크, 결과 비교 같은 자동 판정 기준을 준다. 평가 기준 자체가 올바른지, 실패 로그와 버전이 남는지 확인한다.
4. Independent review 작성 에이전트와 다른 검증 에이전트에게 출처, diff, 테스트 결과를 확인시킨다. 검증이 독립 감사인지, 단순 재요약인지 구분한다.
