ADR 0008 · 2025-Q1

Caching strategy for the hot path

Two-tier 캐시 (request-scoped memo + Redis cache-aside) 로 hot path P99 200ms SLA 회복.

Accepted

P99 -40% (load test)

handoff: Maya

Context

Order placement hot path 가 200ms P99 SLA 침범. 프로파일링 결과 동일 SKU 의 catalog/pricing lookup 반복 (한 request 내 + 같은 user 의 연속 request) 이 60% 차지.

Decision
  • Request-scoped memoization (SKU 키)
  • Redis 공유 캐시 — catalog 30s TTL, price 5s TTL, cache-aside, write-invalidate
  • Negative caching 60s (missing SKU)
Consequences
  • P99 ~40% 감소 (load test 검증)
  • Stale price 최대 5s — product 수용
  • Redis 장애가 hot path 저하 → circuit breaker 로 direct lookup fallback
Alternatives (기각)
  • CDN-level cache — user-key 불가
  • Longer TTL — pricing freshness
  • In-process only — cross-request 도움 없음

ADR 0011 · 2025-Q2

Lazy evaluation on cart pricing

Cart mutation 마다 전체 promo 재계산을 lazy 화. mutation 지연 ~180ms → ~30ms.

Accepted

mutation -83%

handoff: Joon

Context

Cart 변경마다 20+ promo rule 을 eager 평가 → mutation 당 ~150ms CPU. UI 는 line subtotal 만 즉시 노출, full price 는 checkout 에서만 필요.

Decision
  • Mutation 은 pricing_dirty 플래그만 set
  • Full pricing 은 GET /cart/summary 에서 on-demand
  • Per-line subtotal 은 eager 유지
  • Cart version 키 memoization
Consequences
  • Mutation P99 180ms → 30ms
  • Checkout view 지연은 동일 (비용 이동)
  • 새 invariant: cart.total 은 /summary 응답일 때만 신뢰 (API spec 반영)
Alternatives (기각)
  • Cart hash 키 캐시 — cart 가 계속 변해 hit rate 0 근접
  • Mutation 후 async 계산 — checkout race
  • (별도) Rust 재작성 spike — 3x microbench 이긴 했으나 비용 대비 lazy eval 채택

ADR 0014 · 2025-Q3

Index pass on the orders table

200M-row orders 테이블에 partial index 3종 추가. customer history 800ms → 12ms, worker poll 2.5s → 40ms.

Accepted

P99 -98% (worker poll)

handoff: Priya

Context

Orders 는 primary DB 읽기의 80%. EXPLAIN ANALYZE 가 200M row 테이블에서 seq scan 패턴 3종 노출: (customer_id, created_at DESC), (status, updated_at), external_ref lookup.

Decision
  • idx_orders_customer_created partial (deleted_at IS NULL)
  • idx_orders_status_updated partial (status IN ('pending','processing'))
  • idx_orders_external_ref full
  • CONCURRENTLY 빌드, 저트래픽 윈도우
Consequences
  • Customer history 800ms → 12ms P99
  • Worker poll 2.5s → 40ms P99 (secondary alert 제거)
  • Insert ~8% 느려짐 (수용)
  • Disk +12GB (수용)
Alternatives (기각)
  • Customer history MV — freshness 요구사항
  • status 분리 테이블 — invasive vs gain 비율 나쁨
