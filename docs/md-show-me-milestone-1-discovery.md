# md-show-me Milestone 1: Discovery (Scanning + Curation)

## 목적

M1 은 사용자 요청에 맞는 Markdown 후보를 골라낸다. HTML 은 만들지 않는다.

이전 revision 의 *M0 Project Profile* 은 별도 milestone 이었으나, profile cache 폐기 (PRD 4차 정비) 이후 *스캔 규칙 가이드* 만 남아 M1 안으로 흡수됐다. M1 은 이제 *스캔 규칙 + head-preview + agent loop + manifest* 를 한 milestone 에서 다룬다.

Discovery 는 **install-free / network-free** 다 (PRD D11). 어떤 라이브러리도 자동으로 설치하지 않고, 어떤 모델도 자동으로 받지 않는다. 쓰는 도구는 Glob, Grep, LLM 셋뿐이다.

Discovery 는 **sub-agent 안**에서 돈다. main agent 의 context 에는 raw 후보나 preview 가 새지 않고, 최종 manifest 만 올라온다 (parent-context isolation).

## 입력

- 사용자 요청 문자열
- 옵션 `.show-me.toml` (있을 때만)
- 프로젝트 안의 Markdown 파일
- 같은 slug 의 직전 M1 manifest (`.agent-output/show-me/reports/<slug>/manifest.json`, 그 slug 의 `_history/` 가장 최근 폴더) — M5 lazy stale diff 비교용

## 1단계: 스캔 규칙

LLM 이 쿼리 의도에 맞춰 Glob / Grep 으로 즉석 검색한다. 사전 정의된 경로 list 를 순서대로 도는 게 아니다.

LLM 은 training data 에서 `docs/adr/`, `docs/references/`, `docs/meta/` 같은 흔한 markdown 관례를 이미 안다. 별도 명세 없이도 ADR 관련 쿼리면 `docs/adr/` 폴더부터 본다. 비표준 layout (예: `architecture/decisions/`) 도 LLM 이 파일명과 첫 줄 보고 추론.

짧은 redirect-stub 도 인식한다. 루트 MD 가 5줄 이하이고 다른 Markdown 경로를 가리키면, 그 문서를 따라가 실제 source 로 본다.

이전 revision 의 *"읽는 순서 1~6"* numbered list 는 폐기 — LLM 은 sequence 가 아니라 query 로 동작한다. script 도구의 mental model 이었다.

### 우리가 명세하는 것 (LLM 이 training 으로 모르는 부분)

- 보편적 exclude 패턴 (보안·잡음 차단, 아래)
- `.gitattributes` 의 `linguist-generated=true` 자동 합치기
- 옵션 `.show-me.toml` override 형식 (아래)
- output 컨벤션 (`.agent-output/show-me/...`)

### 기본 exclude 휴리스틱

디렉토리 기반 (본문 안 봐도 차단):

- dependency: `node_modules/**`, `.venv/**`, `__pycache__/**`, `*.egg-info/**`
- generated / build: `dist/**`, `build/**`, `coverage/**`, `.agent-output/**`, `.next/**`, `.nuxt/**`, `target/**`, `vendor/**`
- tool worktree: `.claude/worktrees/**`, `.claude/scratch/**`
- `.gitattributes` 의 `linguist-generated=true` 경로

본문 marker 기반: LLM 이 head 5~10줄에서 `AUTO-GENERATED`, `DO NOT EDIT`, "Generated by" 류 marker 또는 frontmatter `auto_generated: true` 를 보면 exclude. strict 키워드 list 는 명시 안 함 — LLM 이 변형도 알아서 인지한다 (예: `AUTO-GEN`, `# WARNING: auto-edit`). 사용자가 자기 marker 정의하려면 `.show-me.toml` 의 `generated_markers` 로 명시.

### Role 분류 휴리스틱

handoff, draft, memory 는 **제외 아니라 role 분류**. 사용자 요청과 맞으면 후보로 올린다.

LLM 이 파일명, 경로, 본문 첫 몇 줄을 보고 role 을 추론한다 (예: `handoff-2026-01.md` → `handoff`, `docs/adr/0042-*.md` → `decision`). 이전 revision 의 strict glob 매칭 표 (`**/handoff/**`, `*.draft.md` 등) 는 폐기 — script 시절 잔재였고, LLM 이 자연어 + naming 관례로 동일한 분류를 더 유연하게 한다.

특이 layout (예: `progress-log.md` 를 handoff 로 쓰기) 은 사용자가 `.show-me.toml` 의 `role_patterns` 에 명시. 그때만 strict 매칭이 들어온다.

### 옵션 config (`.show-me.toml`)

기본적으로 config 는 **필요 없다**. LLM 이 매 호출마다 Glob/Grep + training 관례 + 기본 exclude 휴리스틱으로 동작한다. 비표준 layout 이거나 override 가 필요할 때만 `.show-me.toml` 한 파일을 둔다 (PRD D9).

- 위치: repo root.
- 기본: **gitignored**. 팀이 공유하려면 `.gitignore` 에서 빼고 commit.
- 내용 (모든 필드 옵션):

```toml
trusted_sources = ["docs/adr/**", "docs/references/**", "meta/choices/**"]
role_patterns = [
  { pattern = "handoff.md", role = "handoff" },
  { pattern = "**/handoff/**", role = "handoff" },
  { pattern = "*.draft.md", role = "draft" },
  { pattern = "memory.md", role = "working-memory" },
]
exclude_patterns = ["custom-build/**"]
generated_markers = ["AUTO-GENERATED", "DO NOT EDIT MANUALLY"]
enable_cdn_viz = false
```

`.show-me.toml` 이 있으면 그 값이 default 를 덮어쓴다.

이전 revision 의 `external_sources` (Discord / GitHub PR / ClickUp role 메타데이터) 필드는 폐기됐다. API 연동도 자동 탐색도 안 하는데 죽은 메타였다. 진짜 외부 통합이 들어오면 v2 에서 재도입 검토.

## 2단계: 후보 찾기 (head-preview + agent loop)

기본 흐름은 PRD D10 의 **head-preview + agent loop**.

1. **싼 후보 생성 (rule-based)** — Glob 과 Grep 으로 trusted folder 에서 후보를 모은다. zero install. 후보 수에 상한이 없다 — sub-agent context 가 받을 수 있는 만큼.
2. **싼 preview 추출** — 후보별로 가벼운 preview 를 뽑는다.
   - filename
   - H1 (있으면)
   - frontmatter (있으면, 없으면 skip)
   - 본문 첫 5~10줄
   - 옵션: `##` 수준 헤딩 (grep)
3. **preview 만 보고 LLM 이 큐레이션** — full file 이 아니라 preview 묶음을 LLM 에 넘긴다. LLM 이 "이 N개만 full Read" 라고 정한다. preview 가 작아서 수백 개도 일반 context window 에 들어간다.
4. **필요할 때 agent loop** — preview set 이 부족하다고 LLM 이 판단하면 추가 preview 나 full Read 를 요청한다. preview 자체가 sub-agent context 한도를 넘으면 페이지 단위로 나눠 multi-turn 으로 본다.
5. **큐레이션 set 만 full Read** — 최종 N개 (보통 3~10개, **top-N hard cap 없음**) 만 full Read 해서 M2 / M3 입력으로 넘긴다.

LLM ranking 은 **나중에 따라붙는 단계가 아니라 기본값**이다. 단, LLM 에 들어가는 건 preview 뿐이고 full file 은 아니므로 token cost 에 상한이 잡힌다.

5000+ 파일 규모 repo 도 같은 흐름을 따른다. preview 가 한 turn 에 다 안 들어가면 agent loop 가 페이지 단위로 multi-turn 으로 처리한다. BM25, embedding, 외부 모델은 **어느 규모에서도** 쓰지 않는다 (PRD D4 / D11).

### 후보 신호 (LLM 이 참고하는 hint)

- filename
- title 또는 H1
- frontmatter
- H1 뒤 key-value metadata (frontmatter 가 없을 때)
- abstract
- major headings
- updated date

frontmatter 가 없으면 H1 바로 뒤로 이어지는 `- Key: value` 또는 `- Key: **value**` 줄을 metadata 로 취급한다.

## Abstract

abstract 는 두 군데 중 한 곳에서 온다.

- 문서에 frontmatter `abstract` 가 있으면 그대로 쓴다.
- 없으면 본문 첫 5~10줄을 손대지 않고 그대로 쓴다.

가공을 최소화하는 이유는 단순하다. 잘 쓴 문서는 AI 가 썼든 사람이 썼든 첫 5~10줄에서 자기를 소개한다. 그 자체가 가장 정확한 preview 다.

abstract 갱신은 mtime / size diff 가 트리거. head 5~10줄은 preview 단계에서 어차피 보기 때문에 별도 비용이 없다.

## 매칭 이유 (reason)

후보마다 LLM 이 *왜 이 파일이 쿼리와 관련 있는지* 한 줄 자연어로 적는다.

```
"reason": "title 과 본문이 '리뷰' 워크플로우를 직접 다룸"
"reason": "filename 만 맞음. 본문은 다른 주제."
"reason": "role: handoff 가 쿼리의 'handoff' 키워드와 맞음"
```

정렬은 LLM 이 cluster 판단으로 정한 sources 배열 순서를 그대로 따른다. hard cap 없음.

## Manifest

Manifest 는 사람에게 보여줄 후보 목록과 다음 milestone 의 입력 역할을 같이 맡는다.

```json
{
  "query": "리뷰",
  "created": "2026-05-26T10:00:00+09:00",
  "sources": [
    {
      "path": "docs/example.md",
      "title": "Example",
      "abstract": "문서의 짧은 설명",
      "role": "review",
      "reason": "title 과 본문이 '리뷰' 워크플로우를 직접 다룸",
      "major_headings": ["문제", "결론"],
      "mtime": "2026-05-26T14:30:00",
      "size": 1234,
      "updated": "2026-05-26"
    }
  ],
  "unknown_markdowns": [
    {"path": "docs/scratch/idea.md", "reason": "trusted folder 안인데 role 매칭 안됨"}
  ]
}
```

`unknown_markdowns` 는 trusted folder 안에 있지만 role 분류가 안 된 `.md` 들. M5 lazy check 가 매 호출 채우고, 사용자가 N+3 unknown markdowns triage 패턴으로 분류한다. profile cache 없이 manifest 가 직접 들고 있는다.

`sources[]` 의 길이는 LLM 의 cluster 판단으로 정한다 (보통 3~10). 사용자는 M3 의 query refinement 칩 (N+7) 으로 폭을 조절한다.

긴 excerpt 는 M2 에서 만든다. 명시적 block anchor 가 필요한 드문 케이스는 heading slug (GFM 자동 생성) 로 커버한다.

## 실패와 과다 후보

후보가 하나도 안 잡혀도 제외 조건을 자동으로 풀지 않는다. 대신 검색한 include pattern, 빠진 role 과 pattern, 다시 시도해 볼 만한 요청 예시를 사용자에게 보여준다.

후보가 너무 많아서 preview 단계에서도 LLM context 가 버겁다면, agent loop 가 페이지 단위로 나눠 multi-turn 으로 처리한다. 어떤 규모에서도 BM25, embedding, 외부 모델을 받지 않는다 (PRD D4).

## 완료 기준

- LLM 이 쿼리에 맞춰 Glob / Grep 으로 즉석 검색한다 (사전 path list sequence 없음). 옵션 `.show-me.toml` 이 있으면 override 로 읽는다.
- Glob / Grep 으로 후보를 뽑고 (상한 없음 — sub-agent context 한도 안에서), 싼 preview 를 추출하고, LLM 이 큐레이션하고, 필요하면 agent loop 를 돌고, 최종 set 만 full Read 하는 흐름이 끝까지 굴러간다.
- 어떤 라이브러리도 자동으로 설치하지 않고, 어떤 모델도 자동으로 받지 않는다 (PRD D11).
- Discovery 는 sub-agent 안에서 돌고, main agent 에는 최종 manifest 만 올라온다.
- 후보 개수에 hard cap 이 없다. LLM 이 cluster 를 보고 N (보통 3~10) 을 정한다.
- 후보마다 `reason` 자연어 한 줄이 붙는다 (LLM 이 작성).
- `abstract` 는 frontmatter `abstract` 가 있으면 그것, 없으면 head 5~10줄.
- manifest 는 top-level `query`, `created`, `sources[]`, `unknown_markdowns` envelope 구조.
- manifest 에는 M2 / M3 / M4 가 다시 쓸 heading, mtime, size 가 들어 있다.
- frontmatter 없는 문서도 H1 과 첫 5~10줄로 preview 를 얻는다.
- handoff / draft / memory 는 제외하지 않고 role 로 분류한다.
- 별도 `project-profile.json` cache 파일을 만들지 않는다.
- 외부 협업 도구 (`external_sources`) 필드는 두지 않는다 — PRD 4차 정비에서 폐기됐다.
