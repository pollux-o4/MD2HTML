총평

방향성은 맞다

두 리뷰 모두 Markdown을 저비용 원장으로 두고 HTML을 검토 표면으로 쓰는 역할 분리는 타당하다고 판단했습니다. 특히 LLM이 HTML/CSS를 매번 길게 생성하지 않게 하자는 원칙은 유지할 가치가 있습니다.

현재 설계는 너무 크다

6개 템플릿, 9개 block id, 여러 스크립트, approval MD까지 한 번에 도입하는 계획은 과합니다. MVP는 1개 템플릿, 3개 block id, 최소 frontmatter, review patch 1포맷으로 줄여야 합니다.

깨지는 지점은 환경마다 다르다

comfyui 관점에서는 block drift와 review UX가 먼저 깨지고, discord_chatbot 관점에서는 기존 Discord/RAG/GH PR/ADR 레이어와의 중복이 더 큽니다.

공통 P0

P0 Source of Truth가 불명확하다

원 전략은 worklog MD, approval MD, review MD, HTML이 모두 존재합니다. 이 상태에서는 어떤 파일이 진실이고 어떤 파일이 파생물인지 헷갈립니다.

두 리뷰 모두 approval MD와 HTML은 파생물로 내려야 한다고 봅니다. 직접 쓰는 입력은 원본 Markdown과 review patch로 제한해야 합니다.

P0 block id만으로 stale review를 막을 수 없다

review patch가 file.md#claims를 가리키더라도, 그 사이 block 내용이 바뀌면 코멘트가 낡을 수 있습니다.

comfyui 리뷰는 anchor_quote 또는 hash를 필수로 넣고, 현재 block에 quote가 없으면 fail-loud 처리하라고 제안했습니다.

P0 HTML에서 review patch를 어떻게 작성할지 없다

“HTML 직접 편집 금지”만 있고, 사람이 HTML 화면에서 어떤 UI로 review note를 남기는지가 빠져 있습니다.

두 리뷰 모두 정적 HTML에 textarea/form을 두고 review patch markdown을 클립보드로 복사하는 MVP를 현실적 대안으로 봅니다.

P0 기존 협업 레이어와 중복될 수 있다

discord_chatbot 리뷰는 이미 Discord+RAG, GitHub PR review, ADR이 협업 레이어를 맡고 있는 환경에서는 별도 worklog/review layer가 기능 중복이라고 지적했습니다.

새 시스템은 기존 레이어를 대체하지 말고 ADR, research, handoff 같은 영구 문서의 구조화 강화로 들어가야 합니다.

P1/P2 핵심 우려

심각도 쟁점 리뷰 요지 권장 처리
P1 MVP 과대 6개 템플릿, 9개 block id, 5~6개 스크립트는 첫 사이클에 과합니다. 템플릿 1개, block id 3개, frontmatter 최소 필드, parse/render/review UX부터 시작합니다.
P1 상태머신 미정의 draft, verified, needs-revision, approved, rejected 전이가 정의되지 않았습니다. MVP는 draft, needs-review, approved 3개로 줄입니다.
P1 동시 수정 정책 두 에이전트가 같은 MD나 같은 block을 수정할 때의 정책이 없습니다. MVP에서는 git merge에 맡기되, dangling target은 자동 적용 금지. 필요할 때만 advisory lock을 도입합니다.
P1 효과 측정 부재 토큰 절감, 리뷰 부담 감소, partial edit 성공률이 모두 가설입니다. 3회 실제 사용 후 토큰 사용량, review 시간, stale patch 발생률을 측정합니다.
P2 block 주변 문맥 target block만 읽으면 claims/sources/caveats 일관성이 깨질 수 있습니다. target block + dependency blocks를 컨텍스트 패키지로 넘깁니다.
P2 id 컨벤션 한국어 본문과 영어 block id의 규칙이 명시되지 않았습니다. block id는 영어 kebab-case, 본문은 자유 언어로 고정합니다.

수정된 아키텍처 권고

SSoT

원본 *.md와 review 입력만 source of truth로 둡니다. approval.md, approval.json, approval.html은 build/ 또는 dist/의 파생물로 두고 git 추적 대상에서 제외하는 방향이 더 일관됩니다.

Review SSoT

GitHub PR이 있는 환경에서는 GH PR review thread를 1순위로 둡니다. PR이 없거나 로컬 결재가 필요한 경우에만 *.review.md를 fallback으로 사용합니다.

Block Drift

review patch에는 target만 두지 말고 anchor_quote를 둡니다. 적용 시 quote가 현재 block에 없으면 자동 적용하지 않고 stale/dangling 상태로 표시합니다.

기존 레이어 공존

새 worklog layer를 무조건 만들지 않습니다. 기존 ADR, research, handoff, PR review, Discord/RAG가 있다면 그 레이어를 대체하지 말고 구조화 강화만 적용합니다.

통합 MVP

첫 구현은 “단일 문서 -> 단일 HTML -> review patch 생성” 사이클만 검증한다

여러 Markdown을 모아 approval 문서를 만드는 단계는 아직 넣지 않습니다. 한 문서에 frontmatter와 block id를 적용하고, HTML로 렌더링하고, 사람이 review patch를 생성한 뒤, stale target 검증까지 되는지 먼저 봅니다.

순서 작업 목표
1 문서 1개에 최소 frontmatter와 block id 적용 id, status, exec-summary, findings, next-actions만 검증합니다.
2 parse-md 작성 frontmatter, block id, 중복/누락만 검사합니다. source 없는 strong claim 감지는 MVP에서 제외합니다.
3 render-html + evidence-report 템플릿 1개 LLM이 HTML/CSS를 직접 쓰지 않게 하고, 정적 HTML을 생성합니다.
4 HTML review UI 각 block 옆 코멘트 입력 또는 하단 textarea로 review patch markdown을 만들고 클립보드 복사합니다.
5 stale target 검증 anchor_quote가 현재 block에 없으면 자동 적용하지 않습니다.
6 3회 dogfood 측정 토큰 절감, 리뷰 시간, stale patch, partial edit 실패를 실제로 기록합니다.

원 전략 문서에 반영할 변경 목록

바로 반영

  • §4 디렉터리 구조에서 summaries/, html/을 파생물로 분리하고 build/ 또는 dist/로 이동
  • §5 frontmatter 필수 필드를 id, status 중심으로 축소
  • §5 block id MVP를 exec-summary, findings, next-actions 3개로 축소
  • §7 review patch에 anchor_quote, priority, reviewer, created 추가
  • §7 HTML review 입력 UX를 명시: form/textarea -> patch markdown -> clipboard

설계 결정 필요

  • GitHub PR review thread를 review SSoT 1순위로 둘지 여부
  • *.review.md를 fallback으로 둘지, 항상 생성할지 여부
  • 기존 handoff/ADR/research 문서가 있는 프로젝트에서 새 worklog layer를 금지할지 여부

최종 판단

Approve with changes

원 전략의 핵심 아이디어는 유지합니다. Markdown은 저비용 작업 원장, HTML은 사람이 보는 결재/리뷰 표면, review patch는 사람 피드백을 원본 Markdown의 작은 수정 작업으로 되돌리는 채널이라는 방향은 맞습니다.

다만 지금 문서는 구현 범위가 넓고 source of truth가 흐립니다. 다음 개정에서는 “기존 레이어를 대체하지 않는다”, “approval/HTML은 derived artifact다”, “review patch는 anchor verification을 갖는다”, “MVP는 단일 문서 한 사이클이다”를 명확히 해야 합니다.
