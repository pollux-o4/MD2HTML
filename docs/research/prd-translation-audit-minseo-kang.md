# md-show-me PRD 한국어 audit — 강민서 번역가 시점

## 1. 총평

전체적으로 *정보 전달은 정확*하나, *문장 호흡* 과 *어조* 양면에서 "한국 사람이 처음부터 한국어로 쓴 글" 이라는 느낌은 거의 안 든다. 가장 큰 문제는 두 가지다.

첫째, **영어 한 문장 = 한국어 한 문장** 의 직역식 호흡이 90% 이상 유지된다. 영어 원문은 관계대명사·전치사구로 한 문장을 길게 끌고 갈 수 있지만, 한국어는 같은 길이를 따라가면 어절이 막힌다. 특히 Problem Statement, Solution, Further Notes 의 긴 문장들이 *소리 내어 읽으면* 호흡이 끊긴다. "~하면서, ~하고, ~한다" 식의 영어 분사구문 구조가 그대로 한국어로 옮겨졌다.

둘째, **추상명사 + 기술 영어 음차** 의 비율이 너무 높다. "정직성", "투명성", "큐레이션", "discovery", "surface", "primitive", "transparency", "fallback", "atomic write" 같은 단어들이 한 문장 안에 3–4개씩 들어간다. 기술 문서이긴 하지만 "받아들이는 회의론" 같은 직역어 + "Skepticism we accept" 의 한자어 추상명사는 *원어민이 쓸 한국어가 아니다*. 한국 기술 커뮤니티에서 이런 표현은 보통 "우리도 인정하는 약점" 정도로 풀어 쓴다.

User Story 의 "X 로서, Y 하기를 원한다 — 그래야 Z" 패턴은 **agile 영어 템플릿 ("As a X, I want Y so that Z") 의 1:1 직역**이라 42개가 줄줄이 같은 구조로 나열되면서 한국어 독자에게 단조롭고 부자연스럽게 다가온다. 한국 PRD 에서는 보통 "X 가 Y 할 수 있어야 한다 — Z 위해서" 같은 능동적 어조로 바꿔 쓴다.

Markdown 기술 용어 (`frontmatter`, `Glob`, `Grep`, `hash`, `manifest`) 는 영어 그대로 둬도 자연스럽지만, **추상 개념** (transparency, honesty, skepticism, surface, primitive) 까지 영문 음차하거나 직역하는 건 *과도하다*. 강민서 페르소나라면 추상 개념은 한국어 풀어쓰기로, 기술 명사는 영어로 — 이 경계를 분명히 가져갔을 것이다.

마지막으로 **"~한다는 것이다", "~라는 사실", "~을 가진", "~에 의해"** 같은 직역체 어미가 곳곳에 박혀 있다. 이건 강민서 metric 의 적신호 항목 그대로다.

## 2. 문제 유형 분류

발견된 문제를 6개 카테고리로 묶었다 (심각도는 별도 리스트에서 명시).

**A. 직역체 어미 / 영어 구문 그대로 옮김**
- "~이라는 것이다", "~라는 사실", "~을 가진", "~에 의해", "~함으로써"
- 분사구문 "~하면서, ~하고" 식의 영어식 연결
- 관계대명사 "~인 X" 가 명사 앞에 길게 붙음

**B. 추상명사 남발**
- 정직성, 투명성, 회의론, 정당성, 격리, 강등
- "~성" / "~론" / "~화" 한자어 추상명사가 한 문장에 2–3개 동시 등장
- 한국어로 풀어 쓰면 동사로 변환 가능한 경우가 다수

**C. User Story 템플릿 직역**
- "X 로서, Y 하기를 원한다 — 그래야 Z" 42회 반복
- 영어 As-a/I-want/so-that 템플릿의 기계적 1:1 변환
- 한국어로는 능동·서술 어조가 자연스러움

**D. 영어 음차 과잉 (기술용어 아닌 일반 개념까지)**
- surface, primitive, transparency, baseline, escape hatch, framing, demoted, survived
- 일부는 한국어 대체어가 명확히 존재 (surface → 표면/화면, primitive → 기본 요소, framing → 관점)

**E. 한 문장 너무 김 (3줄 초과)**
- Problem Statement Karpathy 인용 문장, Solution M1 단락, Further Notes Thariq 인용 문장
- 영어 원문 한 문장을 한국어 한 문장으로 그대로 옮긴 결과

**F. 용어 일관성 / 괄호 병기 과잉**
- "head-preview (앞부분 미리보기)", "agent loop (에이전트 루프)", "frontmatter (머리말)", "atomic write (원자적 쓰기)" — 어떤 용어는 병기, 어떤 건 안 함. 기준 불명확
- 같은 개념을 "discovery / 탐색", "큐레이션 / curation" 등 두 표기 혼용

## 3. 개별 문제 리스트

### 사용자가 명시 지적한 3곳 (우선 audit)

**[001] Problem Statement — "서로를 키운다"**

- **위치**: Line 30
- **현재**: "사람과 AI 가 만든 문서 사이의 루프를 망가뜨리는 두 가지 문제가 서로를 키운다."
- **진단**: "서로를 키운다" 는 *positive feedback loop / reinforce each other* 의 직역인데, 한국어 "키우다" 는 보통 *육성, 양성* 의 긍정적 어감이라 *악순환* 맥락에 맞지 않는다. "두 문제가 서로 도와서 더 키워진다" 같은 어색한 해석을 부른다.
- **제안 1**: "사람과 AI 가 만든 문서 사이의 루프를 망가뜨리는 문제가 두 개 있다. 둘은 서로를 부추긴다."
- **제안 2**: "사람과 AI 의 문서 루프는 두 가지 문제 때문에 망가져 있다. 이 둘은 맞물려 서로를 악화시킨다."
- **심각도**: high

**[002] User Story 템플릿 전체 — "X 로서, Y 하기를 원한다 — 그래야 Z"**

- **위치**: Line 97–153 (42개 항목 전체)
- **현재 (예)**: "사용자로서, tool 이 검색 전에 프로젝트 문서 규칙을 이해하기를 원한다 — 그래야 source 문서와 생성물을 헷갈리지 않는다."
- **진단**: agile 영어 템플릿 (As a X, I want Y so that Z) 의 1:1 직역. 한국어로는 *수동적이고 어색*하다. 42개가 줄줄이 같은 구조로 나열되면 *기계 번역체* 로 읽힌다. 특히 "agent 로서, 사용자로서" 가 문두에 오면 한국어 독자는 호흡이 한 번 끊긴 뒤 본문을 읽어야 한다.
- **제안 1 (한국식 능동 어조)**: "검색을 시작하기 전에 tool 이 프로젝트 문서 규칙을 먼저 파악해야 한다. 그래야 source 문서와 생성물이 섞이지 않는다."
- **제안 2 (목적-수단 구조)**: "source 문서와 생성물을 헷갈리지 않으려면, tool 이 검색 전에 프로젝트 문서 규칙을 이해해야 한다."
- **제안 3 (역할 명시가 필요한 경우만 유지)**: 대부분 "사용자로서" 는 빼도 의미가 안 변한다. "agent 로서" 만 살리고 (item 11, 12, 35, 36, 37), 나머지는 주어 생략.
- **심각도**: high (42개 전체에 영향)

**[003] Further Notes §비용 정직성 — "정직성 (Cost honesty)"**

- **위치**: Line 322 (헤딩) + 본문
- **현재**: "### 비용 정직성"

- **제안 1**: "### 비용에 대한 솔직한 인정"
- **제안 2**: "### 비용을 숨기지 않기"
- **제안 3**: "### 비용 trade-off 인정" (영어 trade-off 는 한국 기술 커뮤니티 정착어)
- **심각도**: high

**[004] Further Notes §받아들이는 회의론 — "Skepticism we accept"**

- **위치**: Line 326 (헤딩) + 본문
- **현재**: "### 받아들이는 회의론"

- **제안 1**: "### 우리도 인정하는 약점"
- **제안 2**: "### 반박 가능한 지점들"
- **제안 3**: "### 우리 입장의 한계"
- **심각도**: high

## 4. 선결 결정 권고 (수정 전 사용자가 정해줘야 하는 대원칙)

### Q1. User Story 42개의 템플릿 패턴을 어디까지 한국식으로 재구성할지?

**선택지 A**: 전부 유지 (현재 "X 로서, Y 하기를 원한다 — 그래야 Z" 그대로). 일관성 우선, 영어 PRD 와 1:1 매핑.

**선택지 B**: 전부 한국식 능동 어조로 재작성 ("Y 할 수 있어야 한다. Z 위해서."). 가독성 최우선.

**선택지 C (강민서 추천)**: "사용자로서" 는 대부분 생략, "agent 로서" 처럼 *역할이 실제로 의미가 있는 경우* 만 명시. 어미는 "~해야 한다 / ~할 수 있어야 한다" 능동형으로. 영어 As/I-want/so-that 구조에 갇히지 않음.

**의의**: 이 결정은 42개 항목 전부에 영향. 가장 비싼 결정.

---

### Q2. 추상명사 ("정직성", "투명성", "회의론") 를 모두 풀어쓸지?

**선택지 A**: 헤딩만 풀어쓰고 (예: "비용 정직성" → "비용에 대한 솔직한 인정"), 본문은 한자어 유지.

**선택지 B (강민서 추천)**: 헤딩 + 본문 모두 동사·형용사로 풀어쓰기. "정직성" → "솔직하게 인정한다", "회의론" → "반박 가능한 지점들" / "약점을 인정한다".

**선택지 C**: 그대로 유지. 학술적 톤 우선.

**의의**: PRD 전반 톤을 결정. Further Notes 가장 영향 큼.

---

### Q3. 영어 음차 단어 (surface, primitive, transparency, baseline, escape hatch, framing) 의 한국어 대체 정책?

**선택지 A**: 모두 영어 그대로 유지. 기술 용어로 간주.

**선택지 B (강민서 추천)**: *추상 개념* 은 한국어로 (surface → 화면/표면, primitive → 기본 요소, framing → 관점/시각, escape hatch → 우회 경로), *고유명사적 기술 용어* 는 영어 유지 (discovery, manifest, frontmatter, Grep, Glob, atomic write, hash).

**선택지 C**: 모두 첫 등장 시 괄호 병기 후 영어 유지.

**의의**: 영어-한국어 코드스위칭의 기준선 결정. 일관성 확보의 핵심.

---

### Q4. 영어 한 문장 = 한국어 두 문장 (또는 그 이상) 분할 원칙 — 얼마나 적극적으로?

**선택지 A**: 원문 문장 구조 최대한 보존. 한국어 한 문장이 4–5줄 되더라도 한 문장 유지.

**선택지 B (강민서 추천)**: *3줄 넘으면 무조건 분리*. 영어 dash, 콤마, 분사구문 (-ing) 으로 이어진 부분은 한국어 문장 경계로 끊기.

**선택지 C**: 케이스 바이 케이스 (번역자 판단).

### Q5. 괄호 한국어 병기의 일관성 정책?

**선택지 A**: 모든 영어 기술 용어 첫 등장 시 1회 한국어 병기 (현재 일부 적용).

**선택지 B (강민서 추천)**: *비기술 추상어* 는 한국어로 본문 자체에 풀어쓰기 (병기 X), *기술 명사* 는 영어 그대로 (병기 X 또는 1회만). 현재처럼 "head-preview (앞부분 미리보기) + agent loop (에이전트 루프) Discovery" 식 이중 병기는 가독성 해침.

## 5. 수정 작업 시 따를 원칙 (다음 turn 용)

6. **User Story 능동형 변환**: Q1 결정대로 일괄 처리. "사용자로서" 는 대부분 생략 권장.
